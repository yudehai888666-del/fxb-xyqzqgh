# 学生目标分流与就业情报工作区设计

日期：2026-07-15  
状态：已完成对话设计确认，等待书面规格审阅

## 1. 背景

当前系统按统一的六阶段学生工作流组织业务，第 2 步“目标与情报”已经具备岗位、技能、产业趋势、考试信息和学生技能匹配原型。现有原型的主要不足是：

- 没有在信息采集阶段区分升学导向和就业导向学生。
- 所有学生进入同一套目标分析页面，后续业务差异难以扩展。
- 岗位—技能关系缺少完整的证据治理字段。
- 没有结构化就业市场快照。
- 学生报告直接读取后台最新数据，无法保留当时的审核依据。
- 页面更像录入表单和数据汇总，不足以支持老师解释并形成可追溯结论。

本设计建立“共享学生底座、按主目标加载独立工作区”的架构。本轮重点完成就业路径，升学路径只建立分流入口和引导式页面骨架。

## 2. 已确认的产品决策

1. 新建学生时必须确认一个当前主目标：`升学`或`就业`。
2. 可以设置一个与主目标不同的备选方向，也可以留空。
3. 学生或家长问卷只表达意向，不能直接改变系统业务路径；老师负责正式确认。
4. 首次确认后，目标变化必须填写原因并留下不可修改的变更记录，同时进入重规划流程。
5. 升学型学生使用 A“引导式叙事报告”结构。
6. 就业型学生使用 C“分页工作区”结构。
7. 本轮完成目标分流、就业工作区、可信测试数据和报告快照；不接入真实官方数据源。
8. 日常资料使用结构化数据表，正式学生报告使用不可变 JSON 快照。
9. 老师可以编辑当前分析稿；报告快照生成后不得覆盖，修改必须生成新版本。

## 3. 范围

### 3.1 本轮包含

- 学生主目标、备选方向、首次确认和变更历史。
- 新建学生、学生工作台和第 2 步入口的目标分流。
- 就业型学生六标签工作区。
- 岗位—技能关系的证据、置信度、责任人和复核治理。
- 结构化就业市场快照及图表分布数据。
- 老师可编辑的当前就业分析稿。
- 学生就业情报报告的生成、确认、作废和历史版本查看。
- 报告快照与规划书版本的明确引用关系。
- 现有 `/students/<id>/intelligence-report` 地址的兼容跳转。
- SQLite 现有数据库的幂等轻量迁移。
- 清晰标注的测试数据和学生 ID 7 端到端验证。

### 3.2 本轮不包含

- 真实官方数据源接入、自动解析或定时采集。
- 商业招聘网站批量抓取。
- 院校选择、保研、考研、留学等升学专项数据模型。
- 简历、投递、面试和 Offer 跟踪；这些属于第 5 步“执行复盘”。
- PostgreSQL、对象存储、生产部署和异地灾备。
- 修改现有六阶段顺序。

## 4. 总体架构

系统继续共享以下学生级基础设施：

- 学生身份和基础资料。
- 问卷、材料和免责确认。
- 账号、学生权限和审计日志。
- 老师沟通纪要。
- 文件档案、规划版本和导出。
- 执行复盘与重规划入口。

目标分流服务只负责返回学生当前业务路径，不承载就业或升学领域逻辑：

```text
学生资料与问卷
  → 老师确认当前主目标
    → 升学：引导式升学页面骨架
    → 就业：分页就业情报工作区
  → 目标变化：重规划 + 不可修改的目标变更记录
```

就业领域拆为四个独立单元：

- 就业工作区路由：标签导航、表单提交、权限和页面响应。
- 就业分析服务：匹配度、技能缺口、数据完整度和图表数据。
- 就业快照服务：校验生成条件、事务化冻结数据和版本编号。
- 模板片段：每个标签一个模板片段，避免继续扩大现有单文件报告模板。

## 5. 目标分流模型

### 5.1 `student_goal_profiles`

每名学生最多一条当前目标配置：

| 字段 | 说明 |
|---|---|
| `student_id` | 主键并关联学生 |
| `primary_goal` | `升学`或`就业` |
| `alternate_goal` | 空、`升学`或`就业`，不得与主目标相同 |
| `decision_reason` | 老师确认理由 |
| `confirmed_by` | 确认老师 |
| `confirmed_at` | 首次确认时间 |
| `updated_at` | 当前配置更新时间 |

旧数据库升级后不会为既有学生猜测目标。没有目标配置的学生显示“目标待确认”，确认前不进入具体第 2 步。

### 5.2 `student_goal_changes`

保存首次确认和后续变化：

| 字段 | 说明 |
|---|---|
| `id` | 主键 |
| `student_id` | 学生 |
| `change_type` | `首次确认`或`目标变更` |
| `from_primary_goal` / `to_primary_goal` | 变更前后主目标，首次确认的旧值为空 |
| `from_alternate_goal` / `to_alternate_goal` | 变更前后备选方向 |
| `change_reason` | 必填原因 |
| `replanning_id` | 后续变更必须关联重规划记录；首次确认为空 |
| `changed_by` / `changed_at` | 操作人和时间 |

该表只允许新增，不提供更新和删除业务接口。

### 5.3 目标变更规则

- 新建学生时必须同时创建目标配置和首次确认记录。
- 旧学生在首次进入第 2 步时先完成目标确认。
- 问卷中的倾向作为老师确认依据显示，但不直接写入目标配置。
- 后续目标变更先创建重规划记录，再更新当前目标并写入变化记录。
- 目标变更后，旧路径的报告版本继续可读；旧路径的当前编辑功能停用。

## 6. 就业型学生工作区

就业型学生进入 `/students/<student_id>/employment`。页面保留学生上下文条和六阶段导航，并提供六个可深链接的标签。具体实现可以使用受控的 `tab` 查询参数和独立模板片段，所有写操作使用独立 POST 路由。

### 6.1 标签顺序

1. **目标岗位**：维护第一、第二、第三目标岗位及选择理由。
2. **技能差距**：显示专业—岗位—技能关系、学生当前能力、证据和优先缺口。
3. **市场情报**：显示地区、时间、岗位样本、薪资、学历、经验和热门技能。
4. **证书与考试**：复用已审核考试信息，并转化为学生个人截止日期、负责人和下一步动作。
5. **老师结论**：编辑适合原因、主要风险、行动建议和限制说明。
6. **报告版本**：生成、确认、作废和查看不可变历史版本。

### 6.2 固定工作流

```text
选择目标岗位
  → 评估岗位所需技能
  → 绑定已审核市场快照
  → 确认证书与考试
  → 老师编辑结论
  → 查看缺失项
  → 生成不可变报告版本
  → 确认后供规划书引用
```

页面顶部显示目标类型、备选方向、最近确认时间和依据完整度。生成快照按钮始终可见，但条件不完整时只显示明确的缺失项清单，不生成半成品版本。

## 7. 可信职业情报模型

### 7.1 岗位—技能关系治理

保留现有 `job_skill_links`，复用 `evidence_note` 和 `source_url`，并增加：

- `source_id`：关联受控情报来源。
- `confidence_level`：`高`、`中`或`低`。
- `sample_size`：证据样本量；无法量化时为零并在限制说明解释。
- `last_verified_at`：最近验证日期。
- `next_check_at`：下次复核日期。
- `owner_user_id`：维护负责人。
- `reviewer_user_id`：审核人。
- `status`：`草稿`、`待审核`、`已发布`、`已退回`或`已过期`。
- `limitation_note`：证据边界和不可推断内容。

现有关系在迁移后默认为`草稿`，必须补齐治理字段并审核后才能进入新报告。岗位、技能实体已发布不代表它们之间的关系已审核。

### 7.2 `employment_market_snapshots`

一条记录代表某岗位、地区和统计周期的一次结构化市场观察：

| 字段组 | 内容 |
|---|---|
| 观察范围 | 岗位、地区、周期开始和结束日期 |
| 总量 | 观察到的岗位数量、实际样本量 |
| 薪资 | 最低值、中位值、最高值、币种和计薪周期 |
| 证据 | 来源、来源页面快照、证据摘要 |
| 治理 | 数据分类、负责人、审核人、状态、下次复核时间 |
| 局限 | 样本偏差、重复岗位、口径差异等限制说明 |

`data_classification` 只能为`测试数据`或`真实数据`。数据结构为下一轮真实来源接入预留`真实数据`，但本轮录入界面和测试夹具只创建`测试数据`，不能由用户自行把测试记录改标为真实记录。

市场快照在发布前可以编辑；发布后内容不可修改，修正时创建新快照并将旧记录标记为过期。

### 7.3 `employment_market_breakdowns`

保存图表所需的分布明细：

- `snapshot_id`
- `dimension_type`：`学历`、`经验`、`热门技能`或`地区`
- `label`
- `value`
- `unit`
- `sample_size`
- `sort_order`

统一的分布结构可以支持横向条形图，并保留表格回退，不需要为每种图表新增一张表。

### 7.4 `student_employment_analysis_drafts`

每名就业型学生一份当前分析稿：

- `suitability_summary`：为何适合当前目标。
- `risk_summary`：关键风险和不确定性。
- `action_recommendations`：建议优先级和下一步。
- `limitation_note`：老师对本报告边界的补充说明。
- `updated_by` / `updated_at`。

该记录可以覆盖更新，因为它只是生成报告前的工作稿。每次更新写入现有审计日志。

## 8. 学生报告快照

### 8.1 `student_intelligence_reports`

| 字段 | 说明 |
|---|---|
| `id` | 报告版本 ID |
| `student_id` | 学生 |
| `goal_type` | 本轮为`就业`，为升学报告预留边界 |
| `version` | 学生与目标类型内递增版本 |
| `status` | `待确认`、`已确认`或`已作废` |
| `data_classification` | 根据所含依据计算为`测试数据`或`真实数据` |
| `snapshot_json` | 完整不可变报告内容 |
| `created_by` / `created_at` | 生成人和时间 |
| `confirmed_by` / `confirmed_at` | 确认人和时间 |
| `voided_by` / `voided_at` / `void_reason` | 作废留痕 |

`(student_id, goal_type, version)` 唯一。生成后只允许改变状态和对应状态留痕字段，不允许更新 `snapshot_json`。

### 8.2 快照内容

快照 JSON 必须包含：

- 学生基础信息和当时的主目标、备选方向。
- 第一、第二、第三目标岗位和老师选择理由。
- 当时使用的岗位—技能关系及其证据元数据。
- 学生技能评估、成果证据、匹配度、覆盖度和差距排序。
- 被引用的就业市场快照及全部图表分布。
- 相关产业趋势、证据和治理信息。
- 证书与考试要求、负责人和学生截止日期。
- 老师分析稿的完整文本。
- 系统生成的结论、计算规则版本和生成时间。
- 每条依据的来源、日期、样本量、负责人、审核人和限制说明。

### 8.3 生成门槛

生成报告版本前必须满足：

- 学生当前主目标为就业。
- 至少有一个已发布目标岗位，并明确第一目标。
- 第一目标至少有一个已发布且未过期的岗位—技能关系。
- 第一目标的全部核心技能已完成学生评估；没有证据时允许等级为零，但必须明确“无证据”。
- 第一目标至少绑定一个已发布且未过期的市场快照。
- 被引用市场快照的实际样本量必须大于零；岗位—技能关系无法量化样本时可以为零，但必须解释原因。
- 被引用的趋势、考试和关系记录均已审核且未过期。
- 所有被引用依据都有来源、负责人、审核人、复核日期和限制说明。
- 老师的适合原因、风险、行动建议和限制说明均已填写。

第二、第三目标资料不完整时显示警告，但不阻止以第一目标生成报告。系统在快照中明确记录未完成项。

测试数据不会阻止功能验证，但生成的版本必须醒目标记为“测试数据，仅用于功能验证”，不能设置为学生或家长可见，也不能作为真实业务结论对外导出。

### 8.4 规划书引用

`planning_documents` 增加可空的 `intelligence_report_id`。就业型学生生成规划书时必须选择一个已确认的就业情报报告版本；规划书保存该引用，不自动改用后来生成的新报告。

既有规划版本保持空引用，不做历史推断。

## 9. 升学路径边界

升学型学生进入独立的 A 引导式页面。为了保证分流后没有死路，本轮页面提供：

- 学生和家长问卷中与升学有关的意向摘要。
- 当前考试计划和老师沟通纪要摘要。
- “升学专项模块尚未配置”的明确状态说明。
- 返回学生工作台、进入诊断访谈和形成规划的现有入口。

本轮不建立院校、专业方向、录取概率或申请批次模型。后续应为考研、保研、留学等独立形成设计规格。

## 10. 路由和兼容策略

- `/students/<id>/employment`：就业工作区。
- `/students/<id>/advancement`：升学引导页。
- `/students/<id>/goals/confirm`：旧学生首次目标确认。
- `/students/<id>/goals/change`：通过重规划完成后续目标变化。
- `/students/<id>/intelligence-report`：兼容入口，根据目标跳转。

兼容入口规则：

- 主目标为就业：跳转就业工作区。
- 主目标为升学：跳转升学引导页。
- 目标未确认：跳转目标确认页。

任何目标分流都必须继续经过现有学生级访问权限校验。非当前目标工作区不接受写操作；历史报告版本仍按学生权限提供只读访问。

## 11. 图文表达

就业报告采用现有白色卡片、浅灰背景、蓝色主按钮和约 8px 圆角体系，不引入完全不同的视觉语言。

核心图表：

- 专业 → 目标岗位 → 核心技能关系图。
- 当前水平与要求水平的横向技能差距条形图。
- 学历要求、经验要求和热门技能分布条形图。
- 薪资区间和样本规模摘要。
- 考试与证书时间线。

图表使用服务端计算后的 HTML/CSS 或轻量 SVG，并提供等价表格和文字结论，保证打印、无脚本环境和可访问性。图表旁必须显示来源、统计周期、样本量、负责人、审核状态和限制说明。

## 12. 权限、审计和错误处理

### 12.1 权限

- `admin`：公共资料和来源审核、全部学生访问，以及所有学生级操作。
- 有编辑权限的 `teacher`：确认首次目标、通过重规划变更目标、维护学生分析、生成并确认学生报告。
- `collaborator`：继续由学生级权限控制；本轮就业情报工作区只允许查看，不允许调整目标、维护分析、生成或确认报告。

学生报告确认属于具体学生的业务决策，不等同于公共资料发布，因此不要求管理员逐份确认；报告中引用的公共资料仍必须事先由管理员审核发布。

目标变更、报告生成、报告确认、报告作废、市场快照发布和岗位—技能关系发布均写入审计日志。

### 12.2 错误处理

- 快照生成在单个数据库事务中完成；校验或写入失败时不保留版本号或半成品记录。
- 表单错误返回原页面并保留已填写内容，显示具体字段错误。
- 生成条件不完整时返回结构化缺失项，而不是笼统的“生成失败”。
- 引用的数据在生成前再次检查状态和过期时间，避免页面打开后数据状态已变化。
- 非当前目标路径的写请求返回 409，并引导进入当前路径或重规划，而不是静默写入旧路径。
- 不存在的学生或报告返回 404；没有学生权限返回现有权限层定义的拒绝响应。

## 13. 数据迁移

`schema.sql` 定义新表和新数据库的完整字段；`app/db.py` 中的轻量迁移负责已有 SQLite 数据库：

- 幂等创建目标、市场快照、分析稿和报告版本表。
- 使用 `PRAGMA table_info` 检查后再增加 `job_skill_links` 和 `planning_documents` 字段。
- 现有岗位—技能关系迁移为草稿，不自动赋予来源、审核人或置信度。
- 现有学生不猜测主目标。
- 现有规划版本不猜测情报报告引用。
- 迁移可重复执行，并在测试中覆盖已有旧结构数据库。

## 14. 测试与验收

开发遵循先写失败测试再实现。

### 14.1 自动化测试

- 新建学生必须保存主目标；备选方向不能与主目标相同。
- 旧学生目标待确认，确认后正确分流。
- 后续目标变更必须有原因、重规划记录和审计日志。
- 旧兼容地址按目标正确跳转。
- 非当前目标路径不能写入。
- 岗位—技能关系只在已发布、未过期且治理字段完整时进入分析。
- 市场快照的分布明细能形成稳定图表数据。
- 快照生成门槛返回准确缺失项。
- 快照 JSON 在确认、后台资料变化和生成新版本后保持不变。
- 并发或连续生成版本时版本号唯一递增。
- 测试数据报告不能设置为学生或家长可见，也不能作为真实报告导出。
- 规划书只引用明确选择的已确认报告版本。
- 只读账号不能调整目标、编辑分析或生成报告。
- 旧 SQLite 结构能安全迁移且迁移可重复执行。
